## 1. Контекст проекта

### 1.1 Бизнес-задача
- **Описание проблемы:** Локальные бизнесы (бьюти-сфера, фитнес, кофейни) заинтересованы в кросс-маркетинге (рекламе по бартеру), но у них зачастую нет бюджетов на профессиональных маркетологов, креативщиков и SMM-специалистов. В результате потенциально выгодные коллаборации не случаются из-за сложности в придумывании качественного и конверсионного контента.
- **Ценность:** Создание адаптивной LLM-системы, которая автоматически анализирует профили двух неконкурирующих компаний и генерирует для них готовые идеи и сценарии (Reels, TikTok, посты), решает проблему "чистого листа" и значительно снижает барьер для запуска совместных интеграций. Система учитывает Tone of Voice (ToV) обеих сторон, что делает контент нативным.
- **Критерии успеха:** 
  - Высокий уровень принятия сгенерированных сценариев (Acceptance Rate). 
  - Сокращение времени на согласование креатива между бизнесами с недель до часов.
  - Успешный запуск пилотного MVP с реальными юзерами, метрика «конверсия из генерации идеи в создание контента».

### 1.2 Целевая аудитория и пользователи
- **Users:** Локальный малый бизнес. Владельцы, SMM-менеджеры, администраторы.
- **Нагрузка:** На этапе MVP загрузка оценивается в десятки пользователей в день. В дальнейшем при выходе в прод (SaaS) — сотни одновременных сессий. 
- **Цена ошибки:** Отсутствие соглашения на бартер (сколько денег потеряли при потере клиента на обработку запроса). 

### 1.3 Ограничения и допущения
- **Ограничения:** 
  - Нельзя интегрироваться с источниками оценки компаний (например, Google Карты), оценку можно производить только на моменте разрешения доступа компании к сервису.
  - Фокус только на текстовых сценариях. Парсинг видео и некоторых картинок является ограничением, поэтому даем только идеи для визуала.
- **Допущения:** LLM-модели способны адекватно улавливать Tone of Voice специфичного бизнеса на основе последних 10-20 постов.
- **Безопасность и Репутация:** Безопасность не требует высоких мер, так как анализируется открытая информация о бренде. Для снижения репутационных рисков бот не отвечает на вопросы вне предметной области и гарантирует Brand Safety.

## 2. Архитектура решения

### 2.1 Общая схема
Архитектура строится с применением паттернов **RAG (Retrieval-Augmented Generation)** и **Multi-Agent Systems (MAS)**.
- **Frontend / UI:** Чат-бот (Telegram) или легковесный Web-интерфейс.
- **Orchestrator (Backend):** FastAPI приложение.
- **Data Extractor:** Микросервис парсинга соцсетей (официальные API или инструменты типа Apify).
- **RAG Module:** Векторная БД (VectorDB) для хранения профилей, ToV-гайдов и примеров удачных кросс-маркетинговых креативов.
- **Multi-Agent Pipeline:** Группа взаимодействующих агентов:
  - Brand verification.
  - Analytic Tone of Voice.
  - Matchmaker.
  - Creator.
  - Ad quality.
  - Communication.

### 2.2 Хранилище знаний / документооборот (RAG)

- **Источники:** 
  - Динамические: посты, Reels-описания из соцсетей двух бизнесов.
  - Статические: преобученная модель, база успешных креативов, методологии маркетинга.
- **Предобработка:** Парсинг соцсетей, классификация бренда (норм / стрем), классификация Tone of Voice, анализ контента через классификацию.
- **Индексация:** 
  - **Опция А (Простая, API):** `Qdrant` (или Pinecone): Pinecone suits, small-to-medium businesses needing speed, whereas Qdrant is best for large-scale, cost-effective, self-hosted, or complex filtered search.
- **Обновление:** Данные о бизнесе парсятся "on the fly" при запросе, поэтому "обновление" сводится к кэшированию контекста на 24-48 часов (Redis). 

### 2.3 Многоагентный пайплайн (Агентная система)
Для выполнения задачи используется агентный фреймворк (например, **LangGraph**, **CrewAI**, **AutoGen** или **Phidata**).
Agents
- **Brand verification:** Первичный фильтр бизнеса при регистрации или запросе.
- **ToV:** Берет спаршенные посты, вычленяет уникальные стилистические особенности и классифицирует контент каждого из брендов.
- **Matchmaker** Предлагает 3 наиболее подходящих кандидатов для кросс-маркетинга.
- **Creator** Придумывает логику и расписывает 10 детальных сценариев интеграций.
- **Ad quality** Оценивает 10 предложенных сценариев на конверсионность и жизнеспособность, ранжирует их и выдает пользователю Топ-3.

## 3. Данные и качество знаний

### 3.1 Сбор и предобработка данных
- **Создание набора успешных кейсов:** Ручной (или полуавтоматический) сбор датасета эффективных рекламных кампаний и бартерных интеграций из соцсетей.
- **Предобученные модели:** Использование LLM, дообученных (fine-tuned) на собранных успешных кейсах рекламы, для понимания структуры "пробивающего" креатива.
- **Очистка данных о бизнесе:** Парсинг соцсетей, работа агента на соцсетях бренда с оценкой последних 10-20 постов (на вход идут текст и некоторые картинки), привязка текстов к конкретной медиа-подоплеке поста.

### 3.2 Векторизация и индексирование
- **Индексирование успешных кейсов:** Кодирование базы успешных креативов с помощью моделей эмбеддингов (например, `OpenAI text-embedding-3-small` или открытой `BGE-m3`) с сохранением метаданных о нише (бьюти, еда и т.д.) в векторную БД (Qdrant).

### 3.3 Метрики качества RAG
- **Context Relevance (Релевантность контекста):** Насколько найденный в базе "успешный кейс" релевантен текущей паре бизнесов (попадание в нишу).
- **Faithfulness (Фактологическая точность):** Модель опирается на реальные механики из успешного кейса и не придумывает бизнесам услуги, которых у них нет.
- **Answer Relevance:** Насколько финальный сценарий отвечает первоначальному запросу о кросс-маркетинге.

## 4. Модель и генерация

### 4.1 Выбор LLM и промптинг
Для обеспечения эффективной работы мультиагентной системы требуются модели с продвинутыми способностями к reasoning (логическому выводу) и поддержкой tool calling (вызова внешних функций). Архитектурно рассматриваются несколько стратегий деплоя:

- **Опция 1 (On-Premise / Экономичная):** Использование **Qwen 3** для локального запуска. Обеспечивает высокий уровень приватности данных, независимость от внешних API и минимизацию операционных расходов (OPEX).
- **Опция 2 (Cloud API / Максимальное качество):** Интеграция с проприетарными моделями уровня `GPT-4o` или `Claude 3.5 Sonnet`. В частности, семейство Claude демонстрирует превосходные результаты в креативных задачах, написании естественного текста на русском языке и точном улавливании ToV (Tone of Voice).
- **Опция 3 (Open-weights API / Оптимальный баланс):** Использование мощных открытых моделей, таких как `Llama-3.1-70B` (с быстрым инференсом через Groq API / Together AI) или `Mistral Large 2` для маршрутизации запросов и работы вспомогательных агентов-оценщиков.


### 4.2 Контроль качества агентов и генерации 
- **LLM-as-a-Judge:** Использование сильной LLM (например, Claude Opus) с заранее заданными рубриками для оценки сгенерированных сценариев по критериям: 1-5 баллов за *Креативность*, *Соответствие ToV*, *Реалистичность съемок*.
- **TRAJECT-Bench:** Так как у нас многоагентная система с передачей хода (Agentic workflows), важно оценивать *траекторию* агентов (Trajectory-Aware Evaluation). Мы будем логировать и оценивать не только финальный сценарий, но и:
  - Насколько правильно Агент-Креатор или Парсер использовали "тулы" (инструменты веб-поиска и парсинга API рестула).
  - Корректность передачи хода от Агента-креатора к Агенту-маркетологу (Trajectory Satisfaction). Это позволяет находить "узкое горлышко" (слабого агента).
- **Метрики отсева галлюцинаций (RAGTruth / GlobalQA):** Проверка сценариев на полное отсутствие галлюцинаций относительно несуществующих услуг бизнесов.
